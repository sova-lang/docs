# Internationalisation with `strix/tongue`

`strix/tongue` is Strix's translation layer. It is intentionally
small: one install call seeds a message table, one helper resolves a
key for the active locale, and one setter swaps the locale at
runtime. The active locale is a `@reactive` field, so any view or
effect that reads `t(...)` automatically re-renders when the user
switches language.

## Installation

```toml
[dependencies]
"strix/tongue" = "0.1"
```

```bash
sova install
```

```sova
import "strix/tongue" using *
```

## Message tables

Tongue's message table is a Sova map (or, on the JavaScript side, a
plain object — the type is `any`) shaped
`{ locale: { key: "string" } }`. Keys can be nested through dot
notation, with each segment walking one level deeper into the table:

```sova
let messages = {
    "en": {
        "greeting":   "Hello, {name}!",
        "nav": {
            "home":     "Home",
            "profile":  "Profile",
        },
        "cart": {
            "items":    "{count} items in your cart",
            "checkout": "Check out",
        },
    },
    "de": {
        "greeting":   "Hallo, {name}!",
        "nav": {
            "home":     "Start",
            "profile":  "Profil",
        },
        "cart": {
            "items":    "{count} Artikel in deinem Warenkorb",
            "checkout": "Zur Kasse",
        },
    },
}
```

Looking up `"nav.home"` walks `nav` → `home` and returns the string;
a missing key (no entry in the active locale, no entry in the
fallback locale) falls back to the raw key string, so an
untranslated `t("nav.home")` renders as `"nav.home"` instead of an
empty space — easy to spot at a glance.

## Installing the table

`Tongue(locale, fallback, messages)` seeds the active locale, the
fallback locale, and the message table. Call it once during startup,
typically from a plugin:

```sova
func Tongue(locale: string, fallback: string, messages: any)
```

```sova
let tonguePlugin = func(app: any) {
    Tongue("en", "en", messages)
}

createApp(func(): any { return new AppRoot() })
    .use(tonguePlugin)
    .mount(browser.doc().body().handle())
```

The fallback locale is consulted when a key is missing from the
active locale. Setting fallback to the same value as the initial
locale (as above) effectively disables fallback. A more typical
configuration falls back to English regardless of which locale the
user picked:

```sova
Tongue("de", "en", messages)
```

## Translating

`t(key, params)` resolves a key for the active locale, substitutes
any `{name}` placeholders from `params`, and returns the result as a
plain string. Reads of the active locale go through a reactive
getter, so any view, `effect`, or `computed` that calls `t(...)`
re-runs when the locale changes:

```sova
func t(key: string, params: any): string
```

```sova
type Greeting with Composable, Component {
    @reactive userName: string = "World"

    func view(): Composable {
        return Div(class: "hero") {
            H1 { t("greeting", { "name": this.userName }) }
            P  { t("cart.items", { "count": "3" }) }
        }
    }
}
```

When you want no placeholders, pass `none`:

```sova
Button { t("cart.checkout", none) }
```

## Switching locale at runtime

`setLocale(loc)` writes the active locale; `locale()` reads it. Both
hit the same `@reactive` field, so a setter call triggers every
component that depends on `t(...)` (or on `locale()` directly) to
update:

```sova
func setLocale(loc: string)
func locale(): string
```

A typical language switcher uses the `bind` helper from the core
package to mirror the field into a `Select`:

```sova
type LanguageSwitcher with Composable, Component {
    func view(): Composable {
        return Select(bind: bind(
            func(): any { return locale() },
            func(v: any) { setLocale(v as string) },
        )) {
            Option(value: "en") { "English" }
            Option(value: "de") { "Deutsch" }
        }
    }
}
```

Drop a `LanguageSwitcher` anywhere in your app and the moment the
user picks "Deutsch" the entire UI re-renders with the German
strings. No effort on your part beyond `t(...)` and the bind.

## Loading translations lazily

For larger apps you usually do not want to ship every locale's
message table in the initial bundle. Tongue does not lock you into
anything here — `messages` is just an `any`-typed value, so you can
extend it incrementally and re-install when a new locale is fetched.

A common pattern: install the smallest possible table at startup,
then fetch additional locales over a wire call (or a plain
`fetch()`) when the user picks them:

```sova
type Translations {
    @reactive bag: any = { "en": baseEnMessages }
}

let translations = useNest(new Translations()) as Translations

let tonguePlugin = func(app: any) {
    Tongue("en", "en", translations.bag)
}

func switchLocale(loc: string) {
    if !(loc in translations.bag) {
        let table = backend.loadLocale(loc)        // wired backend call
        translations.bag[loc] = table
        Tongue(loc, "en", translations.bag)        // re-install with the new table
    }
    setLocale(loc)
}
```

The `useNest` helper that holds the table is documented under
[Stores (`strix/nest`)](/frontend/stores).

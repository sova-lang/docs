# Project layout

Sova projects are organised around two ideas: packages, which group
related code, and sides, which decide where each piece of code ends up
at compile time. The conventions below are not enforced by the compiler,
but they are what every Sova project converges on after a few iterations.

## The default tree

A typical project looks like this:

```
my-app/
├── sova.toml
├── sova.lock          # generated by `sova install`
├── src/
│   ├── shared.sova    # type definitions visible everywhere
│   ├── backend/
│   │   ├── main.sova  # backend entry, wired routes
│   │   └── ...
│   └── frontend/
│       ├── main.sova  # frontend entry, view tree, reactive state
│       └── ...
└── tests/             # Sova-side tests (optional)
```

You can split files however you like. Sova does not care whether
backend code lives in `src/backend/` or `src/server/`; it cares about
the `on backend|frontend|shared` annotation at the top of each file.

## `sova.toml`

The manifest describes the project. A representative file:

```toml
[package]
name = "my-app"
version = "0.1.0"

[project]
entry = "src/main.sova"
# Optional - defaults to ["src/**/*.sova"]
sources = ["src/**/*.sova"]

[serve]
port = 8080
host = "127.0.0.1"

[wire]
port = 8080
host = "127.0.0.1"
session_secret = "change-me"

[dependencies]
strix = { path = "../strix/core" }
"strix/dom" = { path = "../strix/dom" }
```

Highlights:

- `[project].entry` is the file Sova considers the "starting point" for
  the build. Multiple-file projects still need this hint so the
  compiler knows where to begin.
- `[project].sources` lets you control which files are included.
  Without it, every `.sova` file under `src/` is compiled.
- `[serve]` configures the HTTP server that hosts the frontend during
  development.
- `[wire]` configures the wire transport (port, host, cookie secret).
- `[dependencies]` lists local or registry packages.

## Sides at a glance

Each Sova file starts with a header:

```sova
package myapp/users on backend
```

Three values are valid: `backend`, `frontend`, `shared`. The rules:

- `on backend` code emits only to the Go artefact. It can call other
  backend or shared code freely.
- `on frontend` code emits only to the JavaScript artefact. Same rule
  in reverse.
- `on shared` code emits to both sides. It can only call other shared
  code and the standard library; calls into side-specific code are
  rejected at compile time.

The compiler enforces these rules. A frontend file importing a backend
package is a compile error; the LSP also hides backend-only symbols
from completion in frontend files.

## Multi-package projects

A larger project usually splits into several packages, mirroring the
directory structure:

```
my-app/
├── src/
│   ├── shared/
│   │   ├── package.sova   # package myapp/shared on shared
│   │   └── todo.sova
│   ├── backend/
│   │   ├── package.sova   # package myapp/backend on backend
│   │   └── routes.sova
│   └── frontend/
│       ├── package.sova   # package myapp/frontend on frontend
│       └── view.sova
└── sova.toml
```

Cross-package imports use the package path:

```sova
import "myapp/shared"
import "myapp/backend" using *
```

The `using *` form re-exports every public symbol from the imported
package into the current scope, so you can call `listTodos()` instead of
`backend.listTodos()`.

## Underscore privacy

Inside a package you can prefix any name with a single underscore to
make it package-private:

```sova
let __seq: int = 0          // framework-internal, still cross-package callable
let _config: any = ...      // package-private, not visible to other packages
func _internalHelper() { }
```

Names that start with a single `_` are hidden from other packages. They
do not appear in completion, they do not appear in package-member
listings, and any cross-package reference is a compile error. Double-
underscore prefixes (`__name`) are exempt — they are reserved for
framework-internal helpers that genuinely need to cross package
boundaries (Strix, for example, uses them this way).
